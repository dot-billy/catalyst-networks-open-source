Security Policies

Security groups and firewall rules let you control which nodes can communicate and on which ports. Groups are applied as Nebula groups in node certificates.

Security Groups

A security group is a named collection of nodes within an organization. Groups serve two purposes:

  1. Nebula groups — When a node's certificate is generated, its security group names are embedded as Nebula groups. This allows Nebula's built-in firewall to enforce group-based rules.
  2. Firewall rule targets — Rules can reference security groups as sources or destinations.

Firewall Rules

Each security group can have multiple firewall rules. A rule defines:

FieldDescriptionExample
ProtocolTCP, UDP, ICMP, or Anytcp
Port / RangeSingle port or range443 or 8000-8999
DirectionInbound or Outboundinbound
Source CIDRsIP ranges allowed10.42.0.0/24
Source GroupsSecurity groups allowedweb-servers
Source NodesIndividual nodes allowedmonitoring-01

Example: Web Server Policy

To allow only your monitoring nodes to SSH into web servers:

  1. Create a security group called web-servers
  2. Create a security group called monitoring
  3. Add your web server nodes to web-servers
  4. Add your monitoring node to monitoring
  5. On web-servers, add an inbound rule:
    • Protocol: TCP, Port: 22
    • Source group: monitoring
  6. Add another inbound rule for HTTPS:
    • Protocol: TCP, Port: 443
    • Source CIDR: 10.42.0.0/24 (all mesh nodes)

Assigning Nodes

Nodes can be assigned to security groups in two ways:

  • From the node detail page — Click "Security Groups" to add/remove group memberships
  • From the security group page — Click "Assign Nodes" to add nodes to the group
Note: When you change a node's security group assignments, its certificate should be regenerated for the new groups to take effect in Nebula's firewall.
